# TORQA token proof (public benchmark)

**P75/P77** — Reproducible, product-grade comparison of fixed **natural-language task specs** and 
**baseline code stubs** against **`.tq` surfaces** that must pass **core parse + diagnostics**.

## Product statement (measured, bounded)

> In this benchmark set, TORQA expressed the same workflow intent with substantially fewer estimated tokens than the paired natural-language prompts, while the `.tq` surfaces satisfy strict core validation. Results are scenario-bound and use a deterministic estimator, not live tokenizer APIs.

This is **not** a claim about general intelligence, arbitrary domains, or exact vendor tokenizers.

## What is being measured

- **Per scenario:** `TASK.md` (NL spec), `BASELINE_CODE.txt` (non-TORQA stub), `app.tq` (TORQA surface).
- **Token counts:** deterministic estimator on full file text (same rule as `torqa-token-measure` / P32).
- **Validation gate:** `.tq` is parsed and run through full diagnostics; **failed scenarios stay in the report** 
  with `ok: false` and are **excluded from aggregate averages** (no hiding — see table and JSON).
- **Ratios:** prompt÷TORQA and (prompt+baseline code)÷TORQA; optional IR size vs surface per row.

- **Estimator:** `utf8_bytes_div_4_v1`
- **Method:** Token count := ceil_utf8_bytes(text) / 4 using integer arithmetic (n+3)//4; empty string -> 0; non-empty -> max(1, (utf8_byte_len + 3) // 4).

## What is *not* being measured

- **Not** model reasoning quality, runtime behavior, or security of generated apps.
- **Not** OpenAI/Anthropic exact token counts (no API; no `tiktoken` in this path).
- **Not** competitive benchmarks against other languages — NL and baseline files are **fixed comparators** in-repo.
- **Not** universal savings on every prompt; scenarios are **curated workflow-shaped** examples.

Generated website or UI bundle size is not the headline for this suite; see the flagship compression path (`docs/BENCHMARK_COMPRESSION.md`, `torqa-compression-bench`) for that slice.

## Why token reduction matters here

Smaller, validated surfaces lower **context cost** for humans and tools when the intent is the workflow model, not prose. TORQA keeps **meaning and checks** in the loop: compression is reported **together with** strict validation, not instead of it.

Workflow- and automation-shaped intent: forms, approvals, data pipelines, branching logic, and small multi-step scripts. Each scenario pairs a fixed NL task (TASK.md), a fixed baseline code stub (BASELINE_CODE.txt), and a `.tq` surface that must pass the same core parse + diagnostics.

## Estimator limitations

`utf8_bytes_div_4_v1` approximates subword tokenization with **ceil(UTF-8 bytes / 4)** (non-empty). It is **stable and reproducible** across machines; absolute numbers differ from GPT/claude tokenizers, but **relative** comparisons within this repo are consistent. Do not treat estimates as invoice-grade.

## Stable schema & reproduction

- **Report schema_version:** `2`
- **Canonical JSON:** `reports/token_proof.json` (sorted keys, regenerated by CLI).
- **Scenario manifest:** `examples/benchmarks/token_proof/manifest.json` (ordered list of `{ id, category, relative_dir }`).

### Regenerate

```bash
torqa-token-proof
# or: python -m src.benchmarks.token_proof_cli
```

### Add a new scenario

1. Create a directory under `examples/benchmarks/<your_id>/` with `TASK.md`, `BASELINE_CODE.txt`, `app.tq`.
2. Ensure `app.tq` passes core validation (parse + diagnostics).
3. Append an entry to `examples/benchmarks/token_proof/manifest.json` (`id`, `category`, `relative_dir`).
4. Run `torqa-token-proof` and commit `reports/token_proof.json`, `docs/TOKEN_PROOF.md`, and tests if ids/counts change.

## Field notes (formulas)

- compression_ratio_prompt_per_torqa = prompt_tokens / max(1, torqa_tokens).
- compression_ratio_combined_per_torqa = (prompt_tokens + baseline_code_tokens) / max(1, torqa_tokens).
- average_prompt_token_reduction_percent_vs_torqa = mean over passing scenarios of (prompt_tokens - torqa_tokens) / prompt_tokens * 100.
- baseline_code_tokens: fixed BASELINE_CODE.txt per scenario (approximate non-TORQA implementation).
- ir_to_torqa_ratio > 1 means canonical IR JSON is larger than the .tq surface (expected); NL→TQ compression is the headline claim.
- Failed scenarios are listed with errors; they are excluded from summary averages.
